## YOUR HUNTING GROUNDS

**1. PANEL DATA OPERATORS**

This is the single most fertile ground for bugs in empirical economics code.

- When lag, lead, or first-difference operators are used (e.g., `L.`, `F.`, `D.` in Stata; `_n-1`, `_n+1`; `shift()` in Python/R), ask:
  - Is the data **sorted correctly** immediately before the operator is applied? The sort must include *all* panel identifiers (e.g., unit ID *and* time ID) — a sort on unit and time alone is insufficient if there are multiple observations per unit-year.
  - Is the panel **balanced**? If not, does the code verify that consecutive rows are actually the consecutive time periods for the same unit? Row-indexing (e.g., `_n-1`, `shift()`) assumes the adjacent row is the adjacent period — on an unbalanced panel this assumption silently fails whenever a unit-period observation is missing, with no error or warning.
  - Is the **correct panel identifier** used? For example, differencing across a multi-level panel (e.g., firm × location × year) requires sorting on all relevant keys, not just a subset.
- Quote the sort and operator code directly and trace through what it actually computes.

**2. MERGE AND JOIN INTEGRITY**

- For every merge or join, check:
  - Are the **correct keys** used? Could observations be duplicated because the merge is many-to-many when it should be many-to-one or one-to-one?
  - Is the merge **order** significant? (e.g., sorting before a merge to ensure a specific match when there are ties)
  - Are unmatched observations **silently dropped**? Is this intended and disclosed?
  - After the merge, is the result the expected shape?
- Many-to-many merges are a common source of silent data corruption — flag any that are not explicitly expected and handled.

**3. VARIABLE PROVENANCE**

Variables are often constructed across multiple files. Trace key variables back to their source:

- Find where each key outcome variable, treatment variable, and instrument is **first defined**.
- If it is defined in a cleaning or preparation file, check that file's construction logic. Does the variable mean what the analysis script assumes it means?
- Look for places where a variable is **redefined or overwritten** between its construction and its use in analysis.
- Check for unit-of-observation mismatches: a variable might be constructed at one level (e.g., per group) but used in a regression at a different level (e.g., per individual) without appropriate adjustment.

**4. MISSING VALUE PROPAGATION**

- When a **log transformation** is applied, what happens to zero values? Do they become missing and silently drop observations? Is the resulting sample restriction disclosed?
- When **arithmetic operations** involve missing values, does the result propagate missing silently across observations or variables?
- Are observations dropped silently when a regression includes multiple variables, some of which have missing values (listwise deletion)? For each regression, identify what observations would be dropped and whether this is intended.
- Are there **implicit sample restrictions** created by missing value rules that make the estimation sample differ significantly from the described sample?

**5. FORWARD-LOOKING CONTAMINATION**

This is a subtle but serious class of bug. A variable is contaminated if it is constructed using information that would not have been available at the time it is meant to represent.

- Are any **baseline or pre-treatment variables** constructed using data from the full sample period, including the post-treatment period? (e.g., a "pre-treatment average" calculated over all years rather than only pre-treatment years)
- Are units **classified into treatment or control groups** using information that is realised after the treatment date?
- Are any **control variables** constructed from outcomes or variables that are themselves affected by the treatment?
- In event studies: does any "pre-period" variable inadvertently use post-period values?
- Trace the construction of treatment variables, instruments, and key controls carefully for any use of future information.

**6. TREATMENT–FIXED EFFECT COLLINEARITY**

This is a general and underappreciated identification failure that can be hidden in otherwise-correct-looking code.

- Is the treatment variable (or any key regressor) **perfectly or near-perfectly collinear** with the included fixed effects? If so, the fixed effects absorb the identifying variation and the estimated coefficient is not interpretable as claimed.
- Common cases: a DiD treatment dummy that is perfectly collinear with unit × time fixed effects; a city-level variable in a regression with city × year fixed effects; any time-invariant treatment in a regression with unit fixed effects.
- Check whether the fixed effects structure in the code would mechanically absorb the treatment. If so, the estimated coefficient on the treatment is identified only from residual variation that may not be what the paper claims.

**7. SILENT DUPLICATION**

Duplicate observations inflate sample sizes and deflate standard errors, often without any error message.

- After every merge, reshape, or append, check whether the number of observations is as expected. Unexplained increases in N are a signal of duplication.
- Can the merge keys uniquely identify observations in both datasets? If not, a merge will silently create duplicates.
- After a reshape from wide to long (or vice versa), are there duplicates on the new key?
- Are there observations with identical values on all key variables that should be unique identifiers?

**8. PANEL AND SAMPLE CONSTRUCTION BIAS**

How the panel is assembled can determine what is estimated, independent of the regression model.

- Does the panel construction **select on outcomes**? For example, keeping only units observed in every period selects survivors, which may be correlated with the outcome.
- Are units included in the sample only if they satisfy a criterion that is endogenous to treatment? (e.g., including a firm only if it survived to the end of the period, or including an observation only if an outcome variable is non-missing)
- Is the **extensive margin** (units entering or leaving the panel) handled in a way that could bias estimates? Are zeros for non-appearing units included or excluded, and is this appropriate?
- For difference-in-differences: are control units added to or removed from the sample in ways that depend on post-treatment outcomes?

**9. MATHEMATICAL CORRECTNESS**

Read the code as a mathematician:

- Are **cumulative effects** computed correctly? Verify that the quantity claimed to be cumulative is actually constructed by summing incremental changes, not by summing level effects that are not additive on the claimed scale.
- Are **averages** computed correctly, accounting for the unit of observation? Averaging a group-level variable across an individual-level dataset implicitly weights groups by their size, which may not be intended.
- Are **percentage changes** calculated correctly from log differences?
- Are **index or percentile** calculations applied at the correct level of aggregation?
- Are any statistical transformations applied in an **order that changes the result** (e.g., log before or after dividing; winsorizing before or after computing a ratio)?
- Are **shares or weights** that should sum to one actually summing to one after any sample restrictions?

**10. STAGGERED DIFFERENCE-IN-DIFFERENCES / TWFE HETEROGENEITY**

Two-way fixed effects (TWFE) with staggered treatment timing is one of the most common sources of silent misidentification in recent empirical work.

- Does the paper use a DiD design where different units are treated at different points in time?
- If so, does the code use standard TWFE (i.e., a single treatment dummy with unit and time fixed effects)? If it does, flag this: under heterogeneous treatment effects, TWFE produces a weighted average of treatment effects where some weights can be **negative**, potentially yielding estimates with the wrong sign even when the true effect is positive for every unit.
- Does the paper acknowledge this limitation? Does the code use a heterogeneity-robust estimator (e.g., Callaway-Sant'Anna, Sun-Abraham, Borusyak-Jaravel-Spiess, de Chaisemartin-D'Haultfoeuille)?
- If TWFE is used without adjustment, does the paper at minimum check for pre-trends and provide a decomposition of the TWFE weights?
- Check whether "clean controls" are used — i.e., whether already-treated units are used as controls for later-treated units. This is a known source of bias in staggered DiD.

**11. SPATIAL AUTOCORRELATION**

When identifying variation is geographic, standard clustered or robust standard errors may be insufficient.

- Does the paper exploit variation across geographic units (cities, counties, states, regions)?
- If so, are standard errors corrected for **spatial autocorrelation** — i.e., the possibility that outcomes in nearby units are correlated with each other beyond what unit fixed effects absorb?
- Does the code cluster at a level that is too fine to account for spatial correlation (e.g., clustering at the county level when shocks are regional)?
- For shift-share (Bartik) instruments: are standard errors clustered at the level of the *shares* or the *shocks*? The appropriate level depends on the source of identifying variation — if shocks are national industry trends, standard errors should reflect the limited number of independent shocks, not the number of local units.

**12. STRING MATCHING AND MERGE QUALITY**

When merges are performed on text identifiers (names, place names, industry codes, sector labels):

- Are there **case sensitivity** issues that cause silent non-matches?
- Are there **trailing whitespace or encoding** differences between datasets that prevent correct matching?
- Are **abbreviations or alternate spellings** handled consistently?
- What is the match rate after the merge? A lower-than-expected match rate — even if not flagged as an error — may indicate systematic non-matching on a non-random subset of observations.
- Does the code verify that the merge achieves the expected match rate, or does it proceed silently regardless?

**11. REPRODUCIBILITY**

- Do any operations produce **non-deterministic results** across runs? For example:
  - Sort operations on data with **ties** in the sort key produce arbitrary orderings, and any subsequent row-indexing will be unstable.
  - Random number generation without a set seed.
  - Many-to-many merges where the match order is undefined.
- Would running the code twice from scratch produce the **identical dataset and results**?
- Are there operations whose output depends on the **order of observations** in memory rather than their values?
