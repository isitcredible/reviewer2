## YOUR HUNTING GROUNDS

**1. VARIABLE PROVENANCE**

For each key variable in the paper (outcome, treatment, instrument, key controls), trace it back to its *first definition* in the construction or cleaning files.

- Does the construction match the verbal description in the paper exactly?
- Is the **wrong summary statistic** used? (e.g., a mean when the paper describes a sum or weighted average; a maximum when a minimum was described; a count when a rate was described)
- Is the variable constructed at the **wrong unit of observation**? (e.g., computed at the firm level but used as if individual-level, or vice versa)
- Is the variable constructed using the **wrong denominator or numerator**?
- Is the variable constructed over the **wrong sample scope**? (e.g., computed over all observations when only a specific subgroup was described)
- If the same key variable is defined or used across multiple analysis files, is it defined **consistently** in each? Inconsistent definitions across files are a silent source of non-reproducibility.

**2. COMPLETENESS OF DEFINED PROCEDURES**

Papers often describe multi-case procedures (e.g., "we do X for case A and Y for case B"). Check whether the code implements *all* cases.

- If the paper describes a procedure applying to two or more groups, subsets, or conditions, does the code implement it for *all* of them, or only a subset?
- If a multi-step process is described (e.g., interpolating gaps of length 1 and length 2), does the code implement *every step*, and does each step handle *all* the cases described?
- If a rule is described as applying universally (e.g., "irrespective of location", "for all inventors", "for all periods"), does the code enforce that universality, or does it silently restrict application to a subset?
- Check whether off-by-one errors cause one case to be handled incorrectly (e.g., a range described as inclusive at both ends but implemented as exclusive at one; a rolling window described as covering N periods but computed over N−1).

**3. MERGE AND JOIN INTEGRITY IN CONSTRUCTION**

Data corruption through merging is one of the most common and silent sources of error in empirical pipelines.

- For every merge or join in the cleaning and construction files:
  - What is the **cardinality** of the merge (one-to-one, many-to-one, many-to-many)? Is this correct and intended?
  - Could the merge **silently inflate** the number of observations by creating Cartesian products? If so, all subsequent calculations on that dataset are corrupted.
  - Does the code **assert or verify** the expected merge cardinality, or does it proceed silently regardless of the match outcome?
  - Are unmatched observations **silently dropped**? Is this intended and disclosed?
- After each merge, ask: is the number of observations what would be expected? Flag any cases where the merge logic could produce an unexpectedly large or small dataset.

**4. SORT ORDER SENSITIVITY IN CONSTRUCTION**

Many data construction operations depend silently on the order of observations in memory.

- Are **row-indexing operators** (e.g., `_n`, `_n-1`, `_n+1` in Stata; `shift()` in pandas; `lag()` in R) used in cleaning or construction files?
  - Is the data **sorted correctly** immediately before each such operator is applied, on *all* relevant identifiers (e.g., unit ID and time ID together)?
  - If the panel is **unbalanced** (units missing in some periods), does the code verify that the previous/next row is actually the previous/next time period for the same unit — or does it silently reference the wrong observation?
- Are **first-differences** or **changes** computed by subtracting one row from the previous row? If so, confirm the sort guarantees that consecutive rows are the correct comparison (same unit, adjacent time period).
- Are any variables constructed using a **running sort** (i.e., the output would differ if the data were sorted differently going in)? Flag these as reproducibility risks.

**5. PANEL CONSTRUCTION DECISIONS**

How the panel is assembled determines what is estimated, even before a regression is run.

- Does the panel construction **select on outcomes or survival**? (e.g., keeping only units observed throughout the full sample period selects survivors, which may be systematically different)
- Are units included or excluded based on criteria that are **endogenous to the treatment or outcome**? (e.g., requiring a unit to meet a threshold that is itself affected by the treatment)
- Is the **extensive margin** handled as described? When units enter or leave the panel, are they assigned zeros, treated as missing, or excluded — and does this match the paper's claim?
- If the paper describes imputing or interpolating missing observations, does the code do this for *all* the cases described — all relevant subgroups, all gap types, all time periods — or only a subset?

**6. CROSS-FILE CONSISTENCY**

Replication packages often consist of many separate scripts. Errors arise when variables are used in analysis files with implicit assumptions about how they were constructed that are violated in the cleaning files.

- Is the **same variable** used in multiple analysis scripts? If so, are the analysis scripts all drawing on the same, consistent construction — or could different runs produce different datasets?
- Are there places where an analysis script **assumes a specific sort order** in the input data that is not guaranteed by the cleaning pipeline?
- Are there operations whose output depends on **which cleaning script ran last**, or on file state that is not deterministically set?
- Does the cleaning pipeline produce **identical output across runs**, or are there many-to-many merges, random operations without seeds, or sort-tie ambiguities that introduce variation?

**7. TIME-VARYING VS. TIME-INVARIANT TREATMENT CODING**

Treatment variables are frequently miscoded with respect to whether they vary over time, which silently mis-specifies the identifying variation.

- Does the paper describe a treatment that **varies over time** for individual units (e.g., a firm enters a programme, a region receives a policy, a worker changes employer)? If so, does the code construct treatment as a time-varying indicator — or does it assign a static treatment value based on a single period (e.g., treatment status in year T applied to all years)?
- Conversely, does the paper describe a **fixed treatment group** (e.g., units always treated vs. never treated)? If so, does the code mistakenly allow treatment status to fluctuate across periods, introducing measurement error?
- When treatment is an intensity rather than a binary (e.g., a dosage, a share, a continuous exposure), is it constructed as time-varying where described, and fixed where described?
- Check the construction of any event-time variable: is the event date defined consistently across units? Is there variation in how the event date is assigned across cleaning scripts or datasets that are later merged?

**8. UNDISCLOSED RESTRICTIONS IN CONSTRUCTION**

The paper describes a sample or variable construction process. The code imposes additional, stricter restrictions that are not mentioned.

- Does the code impose **additional conditions** on top of those described in the paper that narrow the sample or the set of observations being processed?
- Are there **hardcoded filters** (e.g., `drop if year < X`, `keep if obs > N`) that have no counterpart in the paper's description of the sample?
- Are any variables **winsorized, trimmed, capped, or floored** in the construction files without disclosure?
- When a procedure is applied to handle special cases (e.g., gaps, outliers, boundary observations), does the code silently expand or contract the scope of that procedure relative to what was described?
